# 4. SYSTEM ANALYSIS

## 4.1 Introduction

System analysis is a major phase in the design and documentation of any deep learning project because it explains how the problem is studied, how the data is handled, how the model interacts with the data, and how the final outputs are interpreted. In an academic report, this section is important because it connects theory with implementation. It does not only describe what model is used, but also explains why the data pipeline has been organized in a particular way, how the image samples are prepared before training, how the quality of data is observed, and how the final outputs are presented in a meaningful form.

In this project, the system under study is an underwater image enhancement system based on deep learning. Its purpose is to improve the visual quality of underwater images that suffer from common degradation problems such as colour distortion, low contrast, reduced sharpness, haze, non-uniform lighting, and limited visibility. Underwater image enhancement is an important problem because underwater images are widely used in marine biology, environmental monitoring, underwater robotics, archaeological exploration, defence applications, and autonomous navigation. However, due to the physical behavior of light in water, raw underwater images usually appear unclear and visually degraded. Therefore, a system that can recover or improve visual quality has significant practical importance.

Underwater scenes are fundamentally more difficult than normal land scenes. When light travels through water, different wavelengths are absorbed at different rates. Red light disappears quickly, which leads to the dominant blue or green appearance of many underwater photographs. In addition, suspended particles cause scattering, which reduces contrast and visibility. Artificial illumination can further introduce uneven lighting, localized glare, or colour imbalance. Because of these factors, underwater images often exhibit multiple distortions at the same time. This means the enhancement problem is not a single-step operation such as brightness adjustment. Instead, it requires a coordinated correction process involving contrast restoration, colour balancing, structural preservation, and perceptual enhancement.

The present project addresses this problem using a lightweight CNN encoder-decoder architecture with a MobileViT-style bottleneck. Even though the model itself is an important part of the overall approach, the success of the project depends heavily on the data pipeline around it. If the dataset is poorly organized, if degraded images are not matched correctly with their targets, if preprocessing is inconsistent, or if results are not interpreted properly, the enhancement model may fail to achieve useful performance. For this reason, the system analysis chapter becomes one of the most important parts of the report.

This chapter studies the complete support structure of the enhancement system through four major areas:

- collection of dataset
- preprocessing of collected dataset
- data visualization
- caption preparation

Each of these areas contributes directly to the quality and credibility of the project. Dataset collection determines what the model learns from and how broadly it can generalize. Preprocessing transforms raw image files into a structured and learning-ready format. Data visualization supports interpretation of degraded inputs, monitoring of model progress, and presentation of enhanced outputs. Caption preparation ensures that the figures and visual evidence generated by the system can be clearly understood in a formal academic report.

The purpose of this chapter is not merely descriptive. It also provides analytical reasoning behind each design choice. For example, it explains why benchmark datasets were selected instead of arbitrary internet images, why paired data is important for supervised enhancement learning, why fixed-size resizing is necessary, why augmentation is used, and why both quantitative and qualitative interpretations of the result are required. Thus, the chapter acts as a bridge between the project objectives, the technical implementation, and the report-ready explanation of the system.

## 4.2 Overview of the Proposed System

## 4.3 Collection of Dataset

Dataset collection is the first and one of the most fundamental stages of the proposed system. A deep learning model can only learn the patterns represented in the data provided to it. If the dataset lacks diversity, contains poor-quality labels, or does not represent the target problem adequately, then the model will not generalize well in practice. In underwater image enhancement, this issue becomes even more important because underwater conditions vary greatly depending on depth, water clarity, suspended particles, lighting conditions, camera type, viewing angle, and environmental composition.

For this project, the data collection strategy is based on using established benchmark datasets rather than manually assembled image folders from unknown sources. Benchmark datasets are preferred because they are widely accepted in research, come with known structure, enable fair comparison with previous work, and reduce uncertainty in data validity. Since the project is meant for academic evaluation, using benchmark datasets also improves the credibility of the results.

The main dataset used for training is the **EUVP dataset**, particularly its paired underwater scenes subset. The acronym EUVP stands for Enhancement of Underwater Visual Perception. This dataset is commonly used for underwater enhancement tasks because it includes underwater scenes with varying levels of degradation and contains paired examples for supervised training.

In the project repository, the EUVP dataset is organized under the following path structure:

- `data/EUVP/Paired/underwater_scenes/trainA`
- `data/EUVP/Paired/underwater_scenes/trainB`
- `data/EUVP/Paired/underwater_scenes/validation`

The role of each folder is as follows:

- `trainA` contains the degraded underwater input images
- `trainB` contains the corresponding target or clean images
- `validation` contains images reserved for validation during training

The presence of paired input and target folders is especially important because the project uses supervised learning. In such a setup, the model learns by comparing its predicted output with a known reference image. This allows the network to optimize its parameters with respect to a meaningful target. Paired learning is often more stable and easier to evaluate than unpaired learning, particularly in educational or controlled project settings.

From the actual dataset counts in the project:

- EUVP `trainA` contains 2185 images
- EUVP `trainB` contains 2185 images
- EUVP `validation` contains 130 images

These counts are important because they confirm that the training folders are balanced. Since the input and target folders contain the same number of images, the chance of filename mismatch is reduced. The validation set of 130 images is separate from the training pairs, which makes it suitable for observing whether the model is learning generalized enhancement behavior rather than memorizing the training samples.

The second major dataset used in the project is **UIEB**, the Underwater Image Enhancement Benchmark. UIEB is widely used because it contains both paired underwater images and difficult no-reference samples. The project expects the following folder structure:

- `data/UIEB/raw-890`
- `data/UIEB/reference-890`
- `data/UIEB/challenging-60`

The significance of this dataset lies in its dual nature. The `raw-890` and `reference-890` folders provide paired samples that support full-reference evaluation using metrics such as PSNR and SSIM. On the other hand, the `challenging-60` folder contains difficult underwater images without references, making it more realistic for perceptual assessment. In this project, the actual counts are:

- UIEB `raw-890`: 890 images
- UIEB `reference-890`: 890 images
- UIEB `challenging-60`: 60 images

The paired UIEB images are valuable because they help determine whether enhancement outputs remain reasonably close to accepted reference images. The challenging no-reference subset is valuable because practical underwater applications often do not provide any ideal ground truth. Therefore, a robust system should also be tested under such conditions.

The third major dataset used for system evaluation is **RUIE**, or the Real-world Underwater Image Enhancement dataset. The repository points to:

- `data/RUIE`

The actual project data contains around 4230 usable images under this directory. RUIE is useful because it includes several different underwater subsets with different degradation characteristics. This diversity allows the system to be tested across scenarios such as strong colour cast, different quality levels, and scene complexity. As a result, RUIE serves as a generalization benchmark rather than a primary paired training source.

From a system analysis point of view, the use of these three datasets offers several advantages:

- EUVP provides a supervised paired learning foundation.
- UIEB supports both reference-based and no-reference evaluation.
- RUIE introduces real-world diversity and cross-dataset testing.

This combination is academically strong because it avoids dependence on a single dataset. If the model were trained and tested only on one source, it might show good internal performance while failing to generalize. By including additional evaluation sets, the project presents a more realistic measure of effectiveness.

The collected datasets also reflect the nature of underwater visual degradation. Across these datasets, one can commonly observe:

- blue or green colour dominance
- low red channel intensity
- poor visibility in distant objects
- reduced image contrast
- lack of fine structural detail
- haze-like scattering
- brightness inconsistency

These characteristics justify the project objective and explain why a generic image enhancement method may not be sufficient. The dataset collection process therefore does more than gather images. It defines the visual problem space within which the system will operate.

Another important analytical point is the separation of data according to system role. The project distinguishes between:

- training data
- validation data
- evaluation data
- paired data
- no-reference data

This separation is essential in machine learning. Training data is used to update model weights. Validation data is used to monitor progress and tune behavior during learning. Evaluation data is used after training to judge generalization. Paired data is useful for supervision and full-reference metrics, whereas no-reference data better reflects real-world deployment where perfect targets do not exist.

The dataset collection stage can also be viewed as a process of problem representation. The chosen datasets collectively represent different aspects of underwater degradation and enhancement difficulty. Because the project uses multiple sources and multiple evaluation modes, the resulting system analysis becomes richer and more defensible.

In conclusion, the dataset collection phase provides the raw material and experimental foundation of the underwater enhancement system. The use of 2185 EUVP training pairs, 130 EUVP validation images, 890 paired UIEB images, 60 challenging UIEB images, and approximately 4230 RUIE images creates a broad and realistic basis for model learning and evaluation. This stage ensures that the system is grounded in meaningful underwater data rather than in narrow or artificial examples.

## 4.4 Need for Dataset Diversity in Underwater Enhancement

An important discussion that naturally follows dataset collection is the need for dataset diversity. In many image processing tasks, the diversity of training data strongly affects generalization. In underwater image enhancement, this is even more critical because underwater environments are highly variable. A system trained only on one style of underwater image may fail when exposed to a different water body, lighting condition, or camera setting.

Dataset diversity matters because underwater degradation is not uniform. Some images suffer mostly from a strong blue cast, while others are dominated by greenish tones. Some scenes have low visibility due to suspended particles, while others appear washed out because of poor lighting. In shallow water, sunlight may create uneven highlights and reflections. In deep water, images may become darker and more monochromatic. Marine scenes themselves also differ greatly, including corals, fish, rocks, sea plants, sediment, man-made objects, and divers.

If the training and evaluation data cover only a small fraction of these conditions, the model may overfit to a narrow distribution. It may appear successful on familiar images but perform poorly on new data. Therefore, dataset diversity contributes directly to:

- improved generalization
- reduced overfitting
- more stable learning
- fairer evaluation
- stronger practical relevance

The datasets used in this project collectively provide different kinds of diversity. EUVP provides paired examples for supervised learning. UIEB offers a mix of paired and challenging images, which helps bridge controlled and realistic conditions. RUIE contributes large-scale variability across real-world subsets. This combination improves the reliability of conclusions drawn from the system.

Diversity is not only about the number of images. It also refers to the variation in:

- colour distribution
- contrast range
- scene composition
- background depth
- water clarity
- illumination pattern
- object type
- distance from camera

In underwater enhancement, such variation is essential because enhancement decisions that help one class of images may harm another. For example, aggressive contrast enhancement may improve a flat low-contrast scene but oversaturate already bright regions in another image. Similarly, strong colour correction may help green-dominant images while making naturally blue scenes look artificial. A diverse dataset helps the model learn balanced transformations rather than extreme corrections.

For the report, this discussion can be used to demonstrate critical understanding of the problem. It shows that the dataset was not chosen only because it was available, but because its structure supports the broader goals of robust underwater enhancement. Including this reasoning strengthens the academic quality of the system analysis chapter.

## 4.5 Preprocessing of Collected Dataset

Once the datasets have been collected, the next step is preprocessing. Preprocessing refers to all operations performed on raw images before they are fed into the model. This is a crucial step because raw datasets are rarely in a form that can be used directly for efficient and stable deep learning. Images may differ in size, file format, orientation, colour representation, or pair availability. If these inconsistencies are not handled properly, the system may face training instability, shape mismatches, incorrect supervision, or reduced generalization.

In the present project, preprocessing is implemented mainly through the dataset loader and the training configuration. The preprocessing stage transforms the collected underwater images into standardized tensor inputs that are compatible with the neural network.

The first step of preprocessing is **directory validation**. Before attempting to load images, the system checks whether the specified dataset directories exist. If an expected directory is missing, the system raises an explicit error. This is a simple but valuable systems-level design practice because it prevents silent failures and saves time during experimentation.

The second step is **file filtering**. The dataset loader only considers valid image files with standard image extensions such as:

- `.jpg`
- `.jpeg`
- `.png`
- `.bmp`
- `.tif`
- `.tiff`

This filtering is important because dataset folders may sometimes contain non-image files such as text metadata, thumbnails, temporary files, or operating system artifacts. Restricting the loader to known image types improves robustness and prevents runtime errors.

The third step is **paired filename matching**. Since the project uses supervised paired training for EUVP, each degraded input image must correspond exactly to a target image. The loader therefore compares the filenames in the input and target directories and retains only the files that exist in both places. If some inputs do not have matching targets, they are skipped and a warning is recorded.

This step is crucial because pairing errors can severely damage the learning process. If the model is shown one degraded image but trained against the wrong target image, the optimization process becomes meaningless. Correct pairing ensures that the model learns a valid transformation from degraded to enhanced appearance.

The fourth step is **image loading and RGB conversion**. Underwater images may originate from different cameras or preprocessing pipelines and can therefore have different internal formats. To make all samples consistent, the system opens each image and converts it to RGB. This guarantees that every sample has exactly three channels, which matches the expectation of the model architecture. RGB standardization also makes downstream processing simpler and more predictable.

The fifth step is **resizing**. The project configuration specifies an image size of 256, meaning that all images are resized to 256 x 256 pixels. This is a common and necessary choice in image-based deep learning systems. It offers several benefits:

- fixed dimensions for batching
- lower memory consumption
- faster training
- consistent input scale
- easier implementation of encoder-decoder layers

However, resizing also involves a trade-off. Reducing images to a uniform moderate resolution may remove some fine-scale details. In underwater enhancement, very small textures and distant structures may become less pronounced after resizing. Despite this limitation, a 256 x 256 resolution is a practical compromise for a project of this scale because it keeps training computationally manageable while retaining enough information for the model to learn colour and structural corrections.

The sixth step is **tensor transformation**. After the image is loaded and resized, it is converted into a tensor representation suitable for PyTorch. This step converts pixel data into numeric arrays that can participate in tensor operations, gradient computation, and optimization. Without this conversion, the image cannot be processed by the neural network.

The seventh step is **data augmentation**. Augmentation increases the effective diversity of the training data without requiring new image collection. In this project, the training dataset uses:

- random horizontal flips
- random vertical flips
- colour jitter applied to the input image

The random flips help the model become less sensitive to orientation and spatial arrangement. This is reasonable because marine objects and underwater textures do not always appear in a fixed viewpoint. Colour jitter is particularly important for underwater enhancement because underwater lighting varies considerably. By perturbing brightness, contrast, saturation, and hue on the degraded image, the system exposes the model to more variation during learning.

One noteworthy design detail is that colour jitter is applied only to the degraded input image and not to the target image. This helps preserve the target as a stable supervision reference. If both input and target were altered randomly, the network would receive inconsistent learning signals. By augmenting only the input side, the system encourages robustness while maintaining training correctness.

The preprocessing pipeline also includes **graceful handling of unreadable files**. If an image cannot be loaded successfully, the system logs a warning and returns a fallback tensor in certain conditions instead of crashing the entire training process. This is useful because real-world datasets can contain occasional corrupt samples. Fault tolerance improves the practicality of the system.

From the configuration file, the preprocessing and training parameters associated with this stage include:

- image size: 256
- batch size: 8
- epochs: 100
- learning rate: 0.0005

These values show that preprocessing is tightly linked to the training setup. Fixed image size affects memory usage, which in turn influences batch size. Batch size affects optimization dynamics and hardware feasibility. Thus, preprocessing and training cannot be seen as separate concerns; they are part of one integrated system.

In summary, preprocessing serves several essential roles:

- it validates data availability
- it standardizes format and dimensions
- it preserves valid input-target pair relationships
- it improves robustness through augmentation
- it ensures compatibility with the neural network

Without preprocessing, the model would receive inconsistent or unusable data. Therefore, preprocessing is not a minor technical detail but a foundational step in the system analysis of the project.

## 4.6 Importance of Preprocessing for Model Stability

It is useful to discuss separately why preprocessing is so important for model stability and final performance. In many student projects, preprocessing is treated as a routine technical step, but in reality it has direct consequences for convergence behavior, metric quality, and visual outputs.

First, preprocessing creates input consistency. Neural networks train more effectively when input samples have consistent structure. If image sizes vary widely, the batching process becomes difficult and the model may require more complex handling. Fixed dimensions allow efficient parallel computation and predictable feature extraction across layers.

Second, preprocessing reduces noise caused by dataset irregularities. Invalid files, inconsistent colour channels, and unmatched image pairs can all act as noise sources. These issues may not always stop execution immediately, but they can weaken training by introducing confusion into the optimization process.

Third, preprocessing supports generalization. Augmentation prevents the model from becoming overly specialized to one exact arrangement of underwater scenes. A model trained on a narrow distribution may produce good results in validation but weak results in deployment. Augmentation broadens the effective training distribution and helps the system learn more robust correction patterns.

Fourth, preprocessing affects interpretability. When all images are resized and normalized consistently, it becomes easier to compare model outputs across datasets and experiments. This consistency strengthens the fairness of the experimental design and makes the generated figures more coherent.

For report writing, including this section is helpful because it shows awareness that preprocessing influences not only implementation convenience but also experimental validity.

## 4.9 Caption Preparation

Caption preparation is the fourth main component of the system analysis chapter. Although it may seem less technical than dataset collection or preprocessing, it is extremely important for academic documentation. In image-based projects, figures often carry a large share of the evidence. If those figures are not labeled and explained properly, the value of the visual results is reduced. Therefore, caption preparation should be considered part of the reporting system rather than an afterthought.

In this project, caption preparation refers to the creation of descriptive labels and figure captions for visual outputs such as sample comparisons, publication panels, validation examples, and selected evaluation images. The repository contains a script for building publication-style composite figures, and the sample labels used there include references to both dataset identity and scene meaning. Examples include:

- `UIEB - fish scene`
- `UIEB - reef scene`
- `UIEB - diver scene`
- `EUVP - red coral`
- `EUVP - squid`
- `EUVP - crabs`
- `RUIE UCCS - blue cast`
- `RUIE UIQS - quality sample`
- `RUIE UTTS - failure case`

These examples reveal several useful principles behind caption preparation in the project.

First, captions identify the **dataset source**. This matters because the reader needs to know whether the image is drawn from the training domain, a benchmark evaluation set, or a challenging real-world subset.

Second, captions often provide a **scene description**. This makes the visual example more memorable and easier to discuss in the report. Instead of referring to a file number, the report can refer to a fish scene, reef scene, diver scene, coral example, or blue-cast example.

Third, captions can highlight the **type of degradation or improvement**. For instance, they may emphasize that the original image suffers from blue dominance, low visibility, or weak contrast, and that the enhanced image shows improved colour restoration or sharper structural detail.

From an academic point of view, a good caption performs multiple roles at once:

- it identifies the figure
- it explains what is being compared
- it guides the reader's attention
- it summarizes the observed improvement
- it links the figure to the surrounding discussion

In the context of underwater image enhancement, captions are especially useful because many improvements are subtle. A reader may notice general enhancement, but a carefully written caption directs attention to specific qualities such as:

- reduced greenish cast
- improved red-tone recovery
- increased foreground-background separation
- clearer marine object boundaries
- enhancement of low-light areas
- remaining artifacts or limitations

It is helpful in the report to use a consistent style for figure captions. A recommended caption format for this project is:

`Figure Number + Dataset Name + Scene Description + Degradation Observation + Enhancement Result`

## 4.10 Best Practices for Preparing Captions in This Project

For a stronger report, the caption preparation subsection can also include best practices followed or recommended in the project:

- Keep captions concise but informative.

## 4.11 Relationship Between System Analysis and the Model Architecture

## 4.12 Limitations Observed During System Analysis

A strong academic report should also acknowledge limitations. The system analysis phase reveals some practical limitations that may influence the interpretation of results.

One limitation is the dependence on fixed-size resizing. While resizing to 256 x 256 makes training efficient, it may reduce fine details and may not fully preserve the original resolution characteristics of some underwater images.

Another limitation is the dependence on filename-based pairing. This works well when dataset organization is correct, but it assumes that the source dataset has been carefully curated. If external datasets are added in the future, pairing errors could become a risk if naming conventions are inconsistent.

A third limitation is that the project is trained primarily on EUVP. Although cross-dataset evaluation on UIEB and RUIE is included, the learned representation may still reflect some bias toward EUVP-style degradation patterns.

A fourth limitation is the interpretive tension between perceptual quality and reference similarity. The model improves no-reference quality metrics clearly, but does not improve every full-reference metric. This does not invalidate the model, but it shows that enhancement quality depends on the chosen evaluation perspective.

Acknowledging these limitations improves the honesty and maturity of the report. It shows that the system has been evaluated critically rather than described only in positive terms.
